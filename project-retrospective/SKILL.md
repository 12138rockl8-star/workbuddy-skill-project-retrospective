# 项目复盘报告生成器

基于对多份优秀复盘文档的深度分析，提炼出通用的复盘逻辑、结构和语言风格，引导用户完成从信息收集到报告产出的全流程，最终生成格式精美的钉钉文档。

## 触发条件

当用户提到"项目复盘"、"复盘报告"、"项目总结"、"做复盘"、"写复盘"、"月度/季度/年度复盘"等关键词时触发本 Skill。

## 执行流程（严格按顺序执行）

### 阶段一：项目信息收集（必须完成）

触发后，**必须先询问以下信息**，不可跳过直接生成：

#### 1. 基础项目信息
通过 AskUserQuestion 工具向用户提问：

**问题1 - 项目名称与类型：**
"请告诉我这是什么项目的复盘？项目名称是什么？是什么类型的复盘（月度/季度/年度/项目结案/事件复盘）？"

**问题2 - 客户关注板块（多选）：**
"该项目的客户最看重哪些板块？你希望在复盘中着重体现？（可多选）"
选项包括：
- 内容运营（视频/图文发布数据、爆款分析、内容矩阵分类）
- 投流效果（DOU+/信息流投放、CPM/CPE/ROI、行业对标）
- 社群运营（粉丝群活跃度、UGC、转化）
- 品牌声量（曝光量、搜索占位、心智植入、品牌话题增长）
- 转化效果（留资、询单、订单线索）
- 客户满意度（合作顺畅度、响应速度、交付质量）
- 粉丝增长（涨粉数据、画像变化）
- 直播运营（进房率、曝光、互动）
- 营销活动（多场活动对比、节点营销、活动SOP）
- 达人合作（达人分层策略、预算效率、合作亮点）
- 竞品分析（竞品图谱、威胁度评估、行业趋势）
- 品牌策略（品牌基因、用户洞察、认知现状诊断）

**问题3 - 复盘周期：**
"这次复盘覆盖的时间范围是什么？（如：2025年Q3、2025年8月、项目全程等）"

#### 2. 材料清单提供

收集完基础信息后，**必须向用户展示材料清单**，请用户收集并提供。根据用户选择的不同关注板块，生成定制化的材料清单。

**通用必选材料（所有复盘都需要）：**
- [ ] 项目整体KPI目标及达成情况数据
- [ ] 复盘周期内的核心时间线（关键节点、里程碑、突发事件）
- [ ] 项目执行团队构成与分工
- [ ] 上一周期数据（用于环比对比）

**按关注板块的补充材料清单：**

| 关注板块 | 需要提供的材料 |
|---------|-------------|
| 内容运营 | 发布内容清单（标题+发布日期+数据）；优质内容截图/链接；内容栏目规划；内容矩阵分类及占比 |
| 投流效果 | 投放总消耗、CPM/CPE/ROI数据；投放策略调整记录；优秀投流案例数据；行业CPE/CPM参考值 |
| 社群运营 | 社群人数变化、活跃度数据；社群活动记录及参与数据；UGC内容截图 |
| 品牌声量 | 总曝光量、搜索指数变化；品牌词覆盖情况；品牌话题增长数据；竞品对比数据 |
| 转化效果 | 留资数量及质量分析；询单/订单数据；转化漏斗各环节数据 |
| 客户满意度 | 客户反馈记录（正面/负面）；沟通时间线；问题解决记录 |
| 粉丝增长 | 粉丝总量及净增数据；粉丝画像数据（年龄/性别/地域）；同行对比数据 |
| 直播运营 | 直播场次及数据（曝光/进房/互动）；直播内容规划；直播切片数据 |
| 营销活动 | 各场活动曝光量/时间/话题矩阵/核心策略/创意动作/关键数据/反思；活动间对比数据 |
| 达人合作 | 达人清单（名称/粉丝量/报价/视频数据/CPE/CPM）；达人分层策略；合作亮点案例 |
| 竞品分析 | 竞品清单及类型分类；竞品核心营销动作；竞品威胁度评估；行业趋势洞察 |
| 品牌策略 | 品牌基因（特点/哲学/人群画像）；用户洞察；认知现状；产品核心功能 |

**向用户的话术：**
"根据你选择的关注板块，我整理了一份材料清单。请你收集以下材料后发给我，材料越完整，复盘报告的质量越高。如果某些材料暂时没有，也可以先提供已有的，我会标注待补充。"

**材料收集判断标准：**
- 用户至少提供了KPI达成数据和核心时间线 → 可以开始生成
- 材料不完整但用户明确说"先按现有的做" → 可以开始生成，缺失部分标注"[待补充]"
- 材料严重不足（只有项目名称） → 提示用户至少提供基本数据

### 阶段二：复盘报告生成

当用户提供了足够的材料后，按照以下结构和逻辑生成复盘报告。

#### 报告核心逻辑（必须遵循）

复盘的整体逻辑线：**肯定成绩 → 科学解释问题 → 给出可执行的优化方案 → 展望未来**

这句话要作为报告的开篇"复盘逻辑"陈述，用一句话概括整篇报告的叙事主线。

#### 报告标准结构

```markdown
# 【周期】【项目名称】复盘报告

**复盘周期**：YYYY.MM.DD - YYYY.MM.DD
**整理人**：[姓名]
**适用对象**：项目组全体成员/客户

---

> **复盘逻辑**：[一句话概括本次复盘的叙事主线，格式参考：肯定XX成绩 → 科学解释XX问题 → 给出XX方向的优化方案]

---

## 一、KPI达成情况

### （一）整体数据概览
[用表格或列表呈现核心KPI数据，关键正向数据用绿色加粗标注]

### （二）核心数据解读
[按板块拆解关键数据，每个数据点配归因分析]
- [数据点1]：[数据值] [对比上期变化] → 归因：[原因分析]
- [数据点2]：[数据值] [对比上期变化] → 归因：[原因分析]

## 二、核心运营解读

### （一）[客户最关注的板块1]
[详细分析，包含：执行手段、数据表现、解决问题、案例拆解]

#### 优质案例分析
[选择1-3个典型案例深度拆解]
- **案例名称**：[名称]
- **核心数据**：[完播率/播放量/互动率等]
- **分析&归因**：[为什么表现好/不好]
- **Learning**：[可复用的经验]

### （二）[客户最关注的板块2]
[同上结构]

### （三）[其他板块...]
[根据客户关注度排列详略]

## 三、执行中的策略调整
[记录复盘周期内的关键策略变化，包括：为什么调整、怎么调整、调整效果]

## 四、问题点及优化方向

| 问题 | 归因分析 | 优化方向 |
|------|---------|---------|
| [问题1] | [根本原因] | [具体可执行的解决方案] |
| [问题2] | [根本原因] | [具体可执行的解决方案] |

## 五、下阶段规划

### （一）目标设定
[明确下阶段的核心KPI目标]

### （二）行动计划
| 时间 | 内容 | 目标 | 负责人 |
|------|------|------|--------|
| [时间] | [具体行动] | [预期目标] | [负责人] |

## 六、总结与展望
[用2-3句话总结本阶段成果，展望下阶段方向]
```

#### 结构弹性调整规则

根据复盘类型调整结构侧重：

| 复盘类型 | 结构侧重 | 特殊模块 |
|---------|---------|---------|
| 月度复盘 | KPI达成 + 运营解读 + 下月规划 | 月度数据概览、下月排期表 |
| 季度复盘 | KPI达成 + 运营解读 + 策略调整 + 问题优化 + 下季规划 | 栏目迭代追踪、前后对比、同行对比 |
| 年度复盘 | 全年回顾 + 里程碑 + 经验沉淀 + 来年规划 | 年度大事记、全年数据趋势、SOP沉淀、阶段性策略总览表、内容矩阵分类、高表现内容五维拆解、营销活动横向对比、达人合作分层策略、竞品全景图谱、品牌基因分析、结构化不足反思、破局方向建议 |
| 项目结案 | 目标回顾 + 成果总结 + 经验教训 + 续约展望 | 项目初衷回顾、路径拆解、续约建议 |
| 事件复盘 | 事件经过 + 问题还原 + 经验验证 + SOP | 关键时间线、前后对比、操作SOP/Checklist |

#### 语言风格要求（必须遵循）

1. **数据驱动**：所有论断必须有数据支撑，避免空洞描述
2. **归因分析**：不只说"是什么"，必须说"为什么"，格式为"归因：[原因]"
3. **Learning提炼**：每个案例分析后必须有"Learning：[可复用经验]"
4. **前后对比**：用表格呈现"优化前 vs 优化后"或"本期 vs 上期"
5. **关键数据标注**：正向数据用 **加粗** 标注，格式如 **100w**
6. **emoji使用**：适度使用emoji增加层次感（如 ✅ 📊 🔥 📍 ⚠️），但不过度
7. **务实直接**：不绕弯子，不说空话，每句话都有信息量
8. **口诀化总结**：如有SOP或方法论，用口诀/顺口溜形式总结便于记忆
9. **行业对标**：核心指标与行业均值对比，凸显优势或差距（如"CPE仅为行业均值的40%"）
10. **✔️/❌标记法**：用✔️标记成功经验，❌标记需改进项，简洁明了
11. **战略矛盾式分析**（年度复盘）：先点出核心矛盾，再给出破局方向
12. **情感化开篇**（年度复盘）：可用情感化语言开篇增强感染力

### 阶段三：输出为钉钉文档

报告生成后，**必须输出为钉钉文档**，使用 dws CLI 创建。

#### 创建流程

1. **将报告内容写入临时 Markdown 文件**：
```bash
# 将完整的 Markdown 内容写入临时文件
# 注意：换行必须是真实换行符，不能用字面量 \n
```

2. **创建钉钉文档**：
```bash
dws doc create \
  --name "【周期】【项目名称】复盘报告" \
  --content-file <临时markdown文件路径> \
  --content-format markdown \
  --format json
```

3. **如需追加内容或分批写入**（内容过长时）：
```bash
dws doc update \
  --node <DOC_ID> \
  --content-file <追加内容的markdown文件> \
  --content-format markdown \
  --mode append \
  --format json
```

4. **获取文档链接**：
从 create 命令返回的 JSON 中提取 `docUrl` 或 `nodeId`，将链接提供给用户。

#### 钉钉文档格式化要点

在 Markdown 中使用以下格式来美化钉钉文档：

1. **表格**：使用标准 Markdown 表格，钉钉会自动渲染
2. **引用块**：使用 `>` 创建重点提示框
3. **分隔线**：使用 `---` 分隔大章节
4. **标题层级**：
   - `#` 文档标题
   - `##` 一级章节（一、二、三...）
   - `###` 二级章节（（一）（二）...）
   - `####` 三级章节（案例分析等）
5. **列表**：使用有序列表和无序列表组织信息
6. **加粗**：关键数据用 `**加粗**`
7. **emoji**：适度使用增加可读性

#### 内容长度处理

如果报告内容超过单次写入限制，按以下策略分批写入：
1. 第一次 `create`：写入文档标题 + 前两个章节
2. 后续 `update --mode append`：逐章追加
3. 每次追加后检查返回状态

#### 输出后确认

文档创建完成后：
1. 向用户展示文档链接
2. 提示用户检查文档格式
3. 询问是否需要调整内容或格式
4. 如需修改，使用 `dws doc update` 或 `dws doc block` 命令进行编辑

## 关键注意事项

1. **不要编造数据**：所有数据必须来自用户提供的材料，缺失数据标注"[待补充]"
2. **客户导向**：报告的详略和侧重必须匹配客户最关注的板块
3. **先肯定后改进**：即使项目有问题，也要先肯定成绩，再科学分析问题
4. **可执行性**：所有优化建议必须是具体可执行的，不能是空话
5. **归因深度**：归因分析要深入到根本原因，不能停留在表面现象
6. **钉钉文档依赖**：本 Skill 依赖钉钉 dws CLI，执行前确认已登录（`dws auth status`）
7. **Markdown换行**：写入钉钉文档时，换行必须是真实换行符，不是 `\n` 字面量
